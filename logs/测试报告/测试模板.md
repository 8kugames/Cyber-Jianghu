# 联调测试模板

**作用**：定义验收标准、日志规范、报告结构。
**使用方式**：每次测试前，按本模板约束，在执行方案中填充具体配置/时长/Agent数量。

---

## 1. 测试目标（必须可证伪）

验证三条核心假设：
- **H1 自主决策**：在决策点上，稳定产出可执行 Action，并对环境产生可归因 StateDiff
- **H2 行为涌现**：区分 LLM 虚假叙事 vs 真实因果涌现（需干预实验 + 统计差异）
- **H3 技术可行性**：满足最低 SLO/成本/稳定性要求，且（如支持）具备可用降级路径

输出模型选型建议（按任务类型 + provider 配置情况，如实输出，不伪造）。

---

## 2. 范围与约束（强约束）

| 约束项 | 说明 |
|--------|------|
| 测试对象 | 由执行方案定义（参考：`cyber-jianghu-agent` + `cyber-jianghu-server`） |
| Agent 配置 | 由执行方案的配置矩阵定义 |
| 角色创建 | 通过 API 创建具有完整背景的新角色（禁止复用旧角色） |
| 配置冻结 | 测试开始后不得动态改配置；如需调整，记录变更点并重新开始 run_id |
| 恶性 bug | 立即中断 → 记录 → 修复 → 回归验证 → 继续 |

---

## 3. 交付物（验收的唯一载体）

| 交付物 | 要求 |
|--------|------|
| 原始日志 | 保留完整链路（见 §5） |
| 指标汇总 | 按 agent_config 分组的关键指标表（含样本量、分位数） |
| 问题清单 | 按严重度/复现条件/trace_id 记录 |
| 测试报告 | 路径 + 节数结构见 §9 |

---

## 4. 部署与运行要求

| 要求 | 说明 |
|------|------|
| Rebuild 可追溯 | 记录镜像 tag / commit hash / 构建时间 |
| 运行隔离 | 各 agent_config 在日志中可明确区分（agent_config / model_id） |
| 配置冻结 | 测试期间配置不变；变更需新 run_id |

---

## 5. 可观测性与结构化日志

**前置门槛**：未满足本节要求的运行结果，视为"不可归因/不可复现"，不计入验收样本。

### 5.1 必须具备的链路字段

| 字段 | 说明 |
|------|------|
| `run_id` | 一次端到端测试运行唯一标识 |
| `trace_id` | 一次任务/行为链路唯一标识 |
| `agent_config` | 由执行方案定义的配置组名 |
| `model_id` | 实际调用的模型（包括 fallback 时的实际模型） |
| `env_version` | 环境版本（部署记录中获取，不重复记录于每条链路） |

### 5.2 必须记录的事件类型（4种核心）

| 事件类型 | 记录内容 |
|----------|----------|
| Action | 决策上下文 + 提交的操作（工具调用/写入/移动/交易/通信等） |
| ToolResult | ok/timeout/error + error_type + latency |
| StateDiff | action 前后状态差分，用于因果归因 |
| Error | 异常分类（不得静默吞掉） |

**推导规则**：Decision / Observation / Proposal 可从 Action + StateDiff 反推，无需独立记录。

---

## 6. 验收口径

**原则**：所有阈值必须在测试开始前冻结于 §9.1，禁止事后改口径。

### 6.1 H1 自主决策

**目标**：在观测到的决策点上，稳定输出可执行 Action，并对环境产生可归因 StateDiff

**决策点处理**：
- 测试期间记录所有决策点实例（从 Action 反推）
- 事后分类：按输入类型/约束条件归纳 D 及 D*

**指标**：
| 指标 | 公式 |
|------|------|
| DecisionCoverage | 完成 Decision 次数 / 触发决策点次数 |
| ActionSuccessRate | tool_status=ok 次数 / 提交 Action 次数 |
| AttributionRate | 有 StateDiff 且可归因次数 / tool_status=ok 次数 |

**失败模式**：ToolFailure / Timeout / InvalidArgs / PermissionDenied / PlanningMismatch / MemoryMismatch / ParseFailure

**通过条件**：对 D* 同时满足 `DecisionCoverage ≥ X1` 且 `ActionSuccessRate ≥ X2` 且 `AttributionRate ≥ X3`
**未通过**：给出 trace_id 列表 + 首要修复方向

---

### 6.2 H2 行为涌现

**目标**：多主体系统在任务集合 T 上出现系统级指标可重复提升，且干预后显著下降

**只接受系统级指标**（禁止仅凭文本叙事）：
- 任务结果：`TaskCompletionRate`、`TimeToGoal`、`ResourceCostPerTask`
- 多主体效率：`ParallelWorkRatio`、`RedundantActionRate`
- 稳定性：`OscillationRate`、`DeadlockRate`

**实验设计**：
- Baseline：正常机制全开
- Intervention：至少 2 个干预维度（如：移除共享记忆 / 限制通信 / 隐藏角色信息 / 禁用工具类别）
- 每组 N 次运行，输出均值/TP50/TP95 + 方差/置信区间

**通过条件**：Baseline 在至少 1 个系统级指标稳定优于 Intervention，且可复现
**判定**：若仅文本叙事但指标差异不成立 → H2 Fail（LLM 虚假叙事）

---

### 6.3 H3 技术可行性

**目标**：达到最低 SLO + 成本上限 + 稳定性要求

**SLO 体系**（冻结于 §9.1）：
| 类别 | 指标 |
|------|------|
| 延迟 | 端到端 TP50/TP95 上限、工具调用 TP95 上限 |
| 可靠性 | trace 错误率/超时率上限 |
| 成本 | 单任务成本上限、tokens TP95 上限 |
| 资源 | 上下文/内存上限（如适用） |

**通过条件**：关键任务集合 T* 全部满足 SLO/上限
**超限处理**：任一关键项长期超限 → H3 Fail（除非明确标注"当前阶段不以该项为验收目标"）

---

## 7. 模型选型与 Fallback

### 7.1 路由表格式（可观测触发）

`Trigger(可观测信号)` -> `Model(实际 model_id)` -> `SuccessMetric(改善指标)`

### 7.2 Trigger 最小集合

| Trigger | 条件 |
|----------|------|
| 规划失败 | PlanningMismatch 比例 > 阈值 |
| 工具错误 | ToolFailureRate / InvalidArgs > 阈值 |
| 延迟超限 | 端到端 TP95 > SLO |
| 成本超限 | CostPerTask / Tokens TP95 > 上限 |
| 上下文逼近 | ContextUsage > 阈值 |
| 可靠性 | TimeoutRate / ErrorRate > 阈值 |

**阈值冻结**：各 Trigger 的阈值需在 §9.1 冻结。

### 7.3 Fallback 如实规则

| 情况 | 要求 |
|------|------|
| 已配置 fallback | 输出每个 Trigger 对应的实际 fallback model_id，验证指标改善 |
| 未配置 fallback | 标注 `Fallback: Not Configured / Not Supported`，不得伪造 |

---

## 8. 问题处理原则

| 类别 | 处理 |
|------|------|
| 监控周期 | 由执行方案定义（建议 5-30min，根据测试时长调整） |
| 角色死亡 | 记录 trace_id + 死亡原因 + 当时模型/配置，通过 API 转生并创建新角色 |
| 恶性 bug | 关键流程无法推进 / 数据损坏 / 大面积超时导致样本不可用 → 中断 → 修复 → 回归 → 继续 |

- Pass / Fail（按 D*）
- 指标表：DecisionCoverage / ActionSuccessRate / AttributionRate + 失败模式 Top-K

- Pass / Fail

- 干预设计与复现统计（N、方差/区间）

- Pass / Fail（按 T* 与 SLO）
- 延迟 / 错误 / 成本分位数表
- 超限项与处理（含是否触发降级）
